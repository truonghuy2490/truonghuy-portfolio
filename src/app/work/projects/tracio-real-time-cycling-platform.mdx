## Overview

Tracio is a real-time social platform designed for cyclists to track rides, share routes, publish cycling experiences, join challenges, and connect with other riders.

The platform combines GPS ride tracking, route sharing, community blogs, user profiles, challenges, achievements, rankings, and notifications into one cycling-focused ecosystem.

As part of a four-member capstone team at FPT University, I worked as a Backend Developer, focusing on real-time communication, backend services, distributed processing, third-party integrations, and performance optimization.

## System Architecture

Tracio uses a distributed backend architecture where the mobile app and web admin communicate with backend services through an API gateway.

The system integrates Redis for caching, RabbitMQ for asynchronous messaging, MySQL for persistent storage, Firebase for authentication and notifications, AWS S3 for media storage, and Mapbox for map-related features.

![System Architecture](/images/projects/tracio/tracio-architecture.png)

## Database Design

The database was designed around users, routes, blogs, reviews, reactions, media files, shops, bookings, challenges, rewards, notifications, and reports.

## My Contributions

### Backend Development

- Developed backend services using ASP.NET Core Web API with Clean Architecture and MediatR.
- Designed modular business logic to improve maintainability and scalability.
- Built RESTful APIs supporting rides, users, challenges, blogs, achievements, and tracking features.
- Worked with MySQL for relational data storage and backend data operations.

### Real-time Communication

- Built real-time ride tracking features using SignalR to allow cyclists to share live locations during group rides.
- Implemented gRPC services for efficient location synchronization between clients and backend services.
- Used RabbitMQ to process high-frequency GPS updates and reduce bottlenecks during tracking sessions.
- Integrated Redis caching to improve response time for frequently accessed tracking data.

### Third-party Integration

- Configured Firebase Authentication and Firebase Cloud Messaging for user authentication and push notifications.
- Integrated ESMS to support SMS OTP and ZNS OTP verification.
- Connected WeatherAPI to provide real-time weather information for cycling routes.
- Used AWS S3 and AWS Secrets Manager for file storage and secure configuration management.
- Worked with Elasticsearch to support search-related features.

## Challenges and Learnings

Tracio was one of my most technically challenging university projects because it required more than standard CRUD operations. The project introduced me to real-time communication, asynchronous messaging, caching, and distributed backend design.

One major challenge was handling frequent GPS updates during active ride sessions. To improve responsiveness and reduce bottlenecks, we combined SignalR for real-time communication, gRPC for efficient location synchronization, RabbitMQ for queued processing, and Redis for low-latency data access.

Another challenge was designing a backend that could support multiple product areas, including route tracking, blogs, challenges, user profiles, shops, notifications, and reports, while keeping the codebase maintainable.

Through this project, I learned how to design backend services for real-time applications, integrate multiple third-party services, and think more carefully about performance, scalability, and maintainability.

## Outcome

Tracio successfully demonstrated how modern backend technologies can be combined to build a responsive, real-time social platform for cyclists.

The project strengthened my understanding of ASP.NET Core backend development, real-time systems, asynchronous messaging, caching, third-party service integration, and distributed backend architecture. It also prepared me for enterprise software development by giving me hands-on experience with more advanced backend patterns beyond traditional CRUD systems.
